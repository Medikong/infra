L3 인프라 구성 대비 현재 로컬/Ansible 구성 Gap 분석

현재 repo의 VMware/Vagrant, Ansible, kubeadm, Kustomize 구성을 기준으로 “이미 갖춘 것”과 “프라이빗 클라우드형 L3 Kubernetes 운영 구성으로 확장하려면 빠진 것”을 정리한 문서입니다.

대상: cloudnative_study 기준 브랜치: feat/local-cluster-infra-migration 논의 목적: 팀 인프라 방향성 점검

요약

기반은 있음

VM 생성, 서버 부트스트랩, kubeadm 클러스터 생성, local registry, Metrics Server, 앱 배포 루프는 이미 갖춰져 있습니다.

운영 모델은 얕음

노드 역할, 배치 정책, Ingress, 관측성, 백업, 인증서, 보안 정책은 아직 실습 최소 구성에 가깝습니다.

L3 플랫폼은 미완

HA control-plane, 데이터 계층 HA, 스토리지 추상화, GitOps, 멀티 클라우드 IaC까지 가야 L3 구성에 가까워집니다.

한 줄로 보면, 현재 구성은 “클러스터를 띄우고 앱을 배포하는 실습 기반”이고, L3 구성은 “장비와 네트워크만 빌리고 그 위의 운영 생태계를 직접 제어하는 플랫폼”입니다.

현재 있는 구성

로컬 인프라와 서버 부트스트랩

  • VMware/Vagrant 기반 VM 생성
  • control-plane-1, worker-1, worker-2 고정 3노드 구성
  • Ansible로 containerd, kubeadm, kubelet, kubectl, crictl 설치
  • swap 비활성화, 커널 모듈, sysctl, containerd 설정 자동화
  • Vagrant SSH 설정 기반 Ansible inventory 생성

Kubernetes와 앱 배포 루프

  • kubeadm 기반 단일 control-plane 클러스터 구성
  • Calico CNI 설치
  • control-plane VM의 HTTPS local registry 구성
  • Metrics Server 설치 및 kubectl top 확인 루프
  • Kustomize base, overlays/local 직접 apply 방식
영역 현재 상태 의미
노드 control-plane 1대, worker 2대 학습과 기능 검증에는 충분하지만 장애 도메인이나 역할 분리 실험은 제한적입니다.
네트워크 Calico CNI, NodePort 노출 Pod 네트워크는 있으나 Ingress/LB 계층은 없습니다.
이미지 배포 local registry + image build/push target 외부 registry 없이 로컬 클러스터 배포 실험이 가능합니다.
데이터 의존성 PostgreSQL StatefulSet 3개, Kafka StatefulSet 1개 앱 통합 실험은 가능하지만 HA 데이터 플랫폼은 아닙니다.
관측 Metrics Server 리소스 사용량 확인은 가능하지만 로그, 알림, 장기 지표 저장은 없습니다.

L3 구성 대비 빠진 것

Topology 노드 역할 분리

현재 worker는 모두 동일한 성격입니다. PostgreSQL 전용, Kafka 전용, 애플리케이션 전용, edge 전용 노드 같은 역할 모델이 없습니다.

HA Control-plane 고가용성

control-plane이 1대라 API server나 etcd 장애에 취약합니다. 3 control-plane, stacked etcd, control-plane endpoint VIP/LB 구성이 필요합니다.

Network Ingress와 Load Balancer

현재는 NodePort 중심입니다. ingress-nginx, Envoy Gateway, Traefik, Gateway API, MetalLB 또는 kube-vip 같은 진입 계층이 없습니다.

Scheduling 배치 정책

nodeSelector, taint/toleration, affinity, anti-affinity, topologySpreadConstraints가 체계화되어 있지 않습니다.

Data PostgreSQL/Kafka HA

PostgreSQL과 Kafka는 단일 StatefulSet 수준입니다. CloudNativePG, Patroni, PgBouncer, Strimzi 같은 운영형 구성이 없습니다.

Storage 스토리지 추상화

local PV 중심이라 노드 장애와 볼륨 이동에 취약합니다. Longhorn, Ceph, OpenEBS, CSI, VolumeSnapshot 계층이 없습니다.

Ops 관측성과 알림

Metrics Server는 현재 사용량 확인용입니다. Prometheus, Grafana, Alertmanager, Loki, Tempo, OpenTelemetry가 없습니다.

Security 인증서, 시크릿, 정책

cert-manager, External Secrets, Sealed Secrets, Vault, Kyverno, OPA Gatekeeper, NetworkPolicy 운영 규칙이 없습니다.

Delivery GitOps 릴리스 제어

현재는 직접 kubectl apply -k 흐름입니다. Argo CD 또는 Flux 기반의 선언형 배포, drift 감지, 롤백 흐름은 없습니다.

DR 백업과 복구

Velero, MinIO/S3-compatible backup, PostgreSQL 백업/복구, Kafka 복구 절차가 아직 없습니다.

Registry 운영형 이미지 저장소

local docker-registry는 있습니다. 다만 Harbor 수준의 인증, 권한, retention, 취약점 스캔, 프로젝트 분리는 없습니다.

IaC 멀티 클라우드 추상화

현재는 Vagrant/Ansible 로컬 구성이며, AWS/GCP/Azure로 옮길 Terraform 모듈이나 provider-neutral 변수 계약은 없습니다.

팀 논의용 판단 표

질문 현재 답 팀에서 결정할 것
우리가 어디까지 직접 운영할 것인가? 현재는 Kubernetes bootstrap과 앱 배포 루프까지 직접 제어합니다. DB, Kafka, registry, observability, backup까지 직접 운영할지 정해야 합니다.
control-plane HA가 지금 필요한가? 로컬 실습은 단일 control-plane으로 충분합니다. 진짜 서비스형 학습이라면 3 control-plane 구성을 추가할 가치가 있습니다.
Ingress는 무엇으로 할 것인가? 현재는 NodePort 접근입니다. ingress-nginx, Envoy Gateway, Traefik, Gateway API 중 하나를 선택해야 합니다.
데이터 계층을 어디까지 Kubernetes로 가져올 것인가? 단일 PostgreSQL/Kafka StatefulSet 수준입니다. CloudNativePG/Strimzi까지 갈지, EC2 직접 운영 모델을 유지할지 정해야 합니다.
로컬 실습과 클라우드 이식을 어떻게 연결할 것인가? 현재는 Vagrant/Ansible 중심입니다. Terraform으로 substrate를 만들고 Ansible/kubeadm으로 공통 bootstrap을 유지할지 논의해야 합니다.

추천 확장 순서

  1. 노드 토폴로지부터 정리
    Vagrant 노드 정의를 동적으로 바꾸고, control-plane, app, postgres, kafka, edge 라벨을 붙입니다.
  2. 배치 정책 추가
    nodeSelector, taint/toleration, anti-affinity, topologySpreadConstraints를 앱/DB/Kafka manifest에 반영합니다.
  3. Ingress 계층 추가
    로컬에서는 MetalLB + ingress-nginx 또는 Envoy Gateway로 NodePort 의존도를 줄입니다.
  4. 관측성 스택 추가
    Prometheus, Grafana, Alertmanager, Loki를 먼저 붙이고, 이후 OpenTelemetry/Tempo를 고려합니다.
  5. 데이터 계층 운영 모델 결정
    PostgreSQL은 CloudNativePG 또는 EC2 직접 운영, Kafka는 Strimzi 또는 별도 VM 운영 중 하나로 방향을 잡습니다.
  6. 보안/백업/GitOps 보강
    cert-manager, External Secrets, Kyverno, Velero, Argo CD/Flux를 순차적으로 붙입니다.

결론

현재 구성은 학습과 실험을 빠르게 반복하기 좋은 “로컬 kubeadm 실험 플랫폼”입니다. 그러나 L3 수준의 프라이빗 클라우드형 Kubernetes 플랫폼으로 보려면, 단순히 VM 수를 늘리는 것보다 노드 역할, 진입 계층, 배치 정책, 데이터 HA, 관측성, 백업, 보안 정책을 함께 설계해야 합니다.

따라서 다음 논의의 핵심은 “어떤 컴포넌트를 직접 운영할 것인가”입니다. 장비와 네트워크만 빌리고 그 위의 생태계를 직접 관리하려면, 이 문서의 Gap 항목들이 곧 플랫폼 backlog가 됩니다.

작성 기준: 현재 repository의 infra/cluster, k8s/base, k8s/overlays/local 구성. 이 문서는 팀 논의용 초안이며, 실제 구현 우선순위는 운영 목표와 학습 목표에 맞춰 조정합니다.
